# VLESS + REALITY + XTLS Vision（TCP）方案

# 1. 引言

## 1. v2ray/xray生态可用方案

| 方案 | 性能 | 安全性 | 隐蔽性 | 当前推荐度 |
|---|---|---|---|---|
| VLESS + TCP + REALITY + Vision | 很高 | 高 | 高 | 最推荐 |
| VLESS + XHTTP + TLS/REALITY | 中高到高 | 高 | 高 | 新兴推荐，适合替代 WS/gRPC |
| VLESS + TCP + TLS + Vision/XTLS | 很高 | 高 | 中高 | 推荐 |
| Trojan + TCP + TLS | 中高 | 高 | 高 | 推荐 |
| NaiveProxy | 中高 | 高 | 很高 | 隐蔽性很强，但不属于 V2Ray/Xray 核心协议 |
| VMess + WS + TLS | 中等 | 中高 | 中等 | 仍可用，但偏旧 |
| VMess + TCP + TLS | 中高 | 中高 | 中等偏低 | 不太推荐新部署 |
| Shadowsocks 2022 | 高 | 高 | 中等 | 适合轻量代理 |
| Hysteria2 / TUIC | 很高，尤其 UDP | 高 | 中等 | 适合高速场景，但不属于传统 V2Ray 核心协议 |


1. `Hysteria2 / TUIC` 的核心特点
   - 面向高性能网络传输的新一代代理方案，通常基于 QUIC/UDP 构建，因此在高延迟、丢包、跨境链路、移动网络等不稳定环境下，往往比传统 TCP 方案具有更好的速度表现和连接恢复能力；
   - 其中 Hysteria2 更强调带宽利用率、抗丢包和弱网传输体验，TUIC 则同样关注基于 QUIC 的高并发与低延迟传输。
   - 它们的优点是速度快、延迟低、适合视频、下载、游戏等高吞吐场景；
   - 缺点是 UDP 流量在部分网络环境中可能被限制、限速或丢包，隐蔽性通常不如 `NaiveProxy、Trojan TLS` 或 `VLESS REALITY` 这类更接近普通 `HTTPS` 行为的方案。
   - 因此可以概括为：Hysteria2 / TUIC 更偏向“高速传输型方案”，而不是“极致隐蔽型方案”。


2. `Trojan + TCP + TLS` 的核心特点
   - 通过标准 TLS/HTTPS 连接承载代理流量，整体外观接近普通网站访问，因此相比传统 VMess 等方案具有更好的隐蔽性和兼容性；
   - 它的协议设计相对简洁，服务端部署逻辑清晰，安全性主要依赖成熟的 TLS 加密体系，适合有域名、证书和 Web 服务伪装需求的场景。
   - 它的优点是成熟稳定、客户端支持广、隐蔽性较好；缺点是性能通常不如 `VLESS + REALITY + Vision` 这类新方案极致，而且需要正确配置域名、证书和回落网站。
   - 总体可以概括为：`Trojan + TCP + TLS` 是一种成熟稳妥、接近 HTTPS 行为、兼顾安全性与隐蔽性的传统主流方案。

```json
{
    "log": {
        "loglevel": "debug"
    },
    "inbounds": [
        {
            "port": 443, 
            "protocol": "vless",
            "settings": {
                "clients": [
                    {
                        "id": "", // run `xray uuid` to generate
                        "flow": "xtls-rprx-vision"
                    }
                ],
                "decryption": "none"
            },
            "streamSettings": {
                "network": "tcp",
                "security": "reality",
                "realitySettings": {
                    "dest": "", // A website that support TLS1.3 and h2. You can also use `1.1.1.1:443` as dest
                    "serverNames": [
                        ""    // A server name in the cert of dest site. If you use `1.1.1.1:443` as dest, then you can leave `serverNames` empty, it is a possible ways to bypass Iran's internet speed restrictions.
                    ],
                    "privateKey": "", // run `xray x25519` to generate. Public and private keys need to be corresponding.
                    "shortIds": [// Required, list of shortIds available to clients, can be used to distinguish different clients
                        "", // If this item exists, client shortId can be empty
                        "0123456789abcdef" // 0 to f, length is a multiple of 2, maximum length is 16
                    ]
                }
            },
            "sniffing": {
                "enabled": true,
                "destOverride": [
                    "http",
                    "tls",
                    "quic"
                ],
                "routeOnly": true
            }
        }
    ],
    "outbounds": [
        {
            "protocol": "freedom",
            "tag": "direct"
        }
    ]
}
```

- 主要关键词：

| 配置位置 | 参数 | 对应方案组成 |
|---|---|---|
| `"protocol": "vless"` | 入站协议是 VLESS | VLESS |
| `"network": "tcp"` | 传输层使用 TCP | TCP |
| `"security": "reality"` | 传输安全使用 REALITY | REALITY |
| `"flow": "xtls-rprx-vision"` | VLESS 流控模式 | Vision |
| `"port": 443` | 监听 HTTPS 常用端口 | 伪装成常见 HTTPS 入口 |
| `"dest": ""` | REALITY 回落目标 | 非法连接/探测流量的回落网站 |
| `"serverNames": [""]` | 允许的伪装 SNI | 与 `dest` 的证书域名对应 |
| `"privateKey": ""` | REALITY 服务端私钥 | 与客户端 `publicKey` 对应 |
| `"shortIds": [...]` | REALITY shortId | 客户端也要匹配 |

### 3. 服务器分流策略

```sh
# 确认直连 IP
curl -4 https://api.ipify.org
```

# 5. reality代理流量fallback到nginx（sni回落）

如果 Xray/XTLS 需要监听 443，原来网站就不能继续直接占用 443，因此可让 Xray 监听 443，然后把普通网站流量 fallback 到 nginx/Caddy。文档明确列出这种方案：Xray listens on port 443，并通过 Fallbacks 按 SNI 将网页流量回落到 nginx 或 Caddy。

社区参考资料：[sni回落](https://xtls.github.io/document/level-1/fallbacks-with-sni.html)


## 1. 服务器端xray配置文件修改


需要修改的核心参数

```jsonc
"realitySettings": {
  "dest": "www.cloudflare.com:443",
  "serverNames": [
    "www.cloudflare.com"
  ],
  "privateKey": "wN......mY",
  "shortIds": [
    "fb......6"
  ]
}
```


建议改成

```jsonc
"realitySettings": {
  "target": "127.0.0.1:8443",           // 要让普通/失败连接回落到本机 nginx，通常用8443端口
  "serverNames": [
    "yourdomain.com"                    // 需要修改成ip对应的域名，例如 google.com
  ],
  "privateKey": "wN......mY",
  "shortIds": [
    "fb......6"
  ]
}
```

## 2. windows客户端xray配置文件修改

注意：DNS only 会直接返回你的源站 IP，不会把 HTTP/HTTPS 流量经过 Cloudflare 网络代理；而 Proxied 模式下 Cloudflare 会位于访客和源站之间。


## 3. 服务器端nginx配置文件修改

需要修改的核心参数 `listen 443 ssl`

```sh
server {
    listen 443 ssl;
    server_name yourdomain.com;

    ssl_certificate /etc/nginx/key_crt/yourdomain.com.crt;
    ssl_certificate_key /etc/nginx/key_crt/yourdomain.com.key;
    ssl_protocols TLSv1.2 TLSv1.3;
```

改成

```sh
listen 127.0.0.1:8443 ssl http2;
```

如果 Xray REALITY 的 target 指向本机 nginx，nginx 作为“真实 HTTPS 回落网站”时，支持 h2 会更像一个正常现代 HTTPS 站点，也有利于浏览器访问时使用 HTTP/2。

然后执行如下操作

```sh
nginx -t
systemctl restart nginx
systemctl restart xray
```




## 4. SSL证书修改

1. 背景：在使用 `Cloudflare DNS only + Xray REALITY + nginx` 本地回落方案后，浏览器会直接访问源站 nginx，因此不能继续使用只被 Cloudflare 信任的 `Origin Certificate`，需要换成浏览器公开信任的 `Let’s Encrypt` 证书。

2. 在新服务器上，如果 nginx 已经正确监听 80 端口，并且域名 `yourdomain.com` 已经 `DNS only` 解析到该服务器公网 IP，按这个流程通常就可以完成证书申请，并验证自动续期。

3. 关于Certbot和Cloudflare证书区别：
   - Certbot 是一个自动申请、安装和续期 Let’s Encrypt 免费 HTTPS 证书的工具，申请到的证书由公开 CA 签发，浏览器和操作系统默认信任。
   - Cloudflare Origin Certificate 主要用于 Cloudflare 代理节点与源站服务器之间的加密连接，浏览器通常不直接信任；而 Certbot 申请的 Let’s Encrypt 证书适合 浏览器直接访问源站，因此更适合 DNS only 场景。



```sh
# 更新软件包索引
apt update

# 安装 Certbot 主程序
apt install certbot -y

# 安装 Certbot 的 nginx 插件
apt install python3-certbot-nginx -y

# 使用 nginx 插件申请证书，只申请证书，不自动改写 nginx 配置，
# 将`yourdomain.com`替换成自己服务器对应的域名，例如 `google.com`
certbot certonly --nginx -d yourdomain.com

# 测试自动续期流程是否正常
certbot renew --dry-run

# 查看系统中是否存在 Certbot 自动续期定时器
systemctl list-timers | grep certbot
```

- 上述命令组的作用是：通过 Certbot 的 nginx 插件自动完成域名验证并申请证书，同时测试 Let’s Encrypt 证书的自动续期机制是否正常，避免 90 天后证书过期导致网站 HTTPS 访问报错。

申请成功后，nginx 的 HTTPS 配置中使用：

```
ssl_certificate /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;
```

注意，要将`yourdomain.com`替换成自己服务器对应的域名，例如 `google.com`。
